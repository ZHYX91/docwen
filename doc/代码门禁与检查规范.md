# 代码门禁与检查规范

本文档是本项目唯一的代码检查权威指南。所有检查规则以 `pyproject.toml` 为唯一配置来源（Single Source of Truth）。

## 一、门禁入口与执行方式

### 1.1 综合门禁（日常与 CI 使用）
日常开发与 CI 统一使用 `tools/qa.py` 作为唯一入口：

- 快速门禁（默认）：`python tools/qa.py` （包含 Ruff、Pyright 以及非耗时单元测试）
- 全量测试：`python tools/qa.py --suite full`
- 仅运行部分环节：
  - 跳过 Ruff：`python tools/qa.py --skip-ruff`
  - 跳过 Pyright：`python tools/qa.py --skip-pyright`
  - 跳过 Pytest：`python tools/qa.py --skip-pytest`

### 1.2 专项工具（开发调试使用）
- 仅运行类型检查：`python tools/typecheck.py` （无需完整运行测试，适用于专门排查和验证 Pyright 报错）

### 1.3 依赖安装建议
进行开发与门禁检查前，请确保安装了所有验证依赖：
```bash
pip install -e ".[lint,test]"
```

## 二、Ruff 策略

Ruff 分为 Formatter 与 Linter 两部分，均由 `pyproject.toml` 中的 `[tool.ruff]` / `[tool.ruff.lint]` 统一配置。

### 2.1 Formatter

- 入口：`ruff format --check .` (CI 或门禁中) / `ruff format .` (本地格式化)
- 目标：统一代码格式，避免格式差异导致的无意义 diff。我们采用 120 字符行长约束（`line-length = 120`）。

### 2.2 Linter

- 入口：`ruff check .` (CI 或门禁中) / `ruff check --fix .` (本地自动修复)
- 规则集：以基础质量规则为主（E/F/W/I/UP/B/SIM/C4/PIE/RUF/T20）。
- 关键忽略项：
  - `E501`：行长由 formatter 统一处理，不用 linter 再次报错。
  - `RUF001/RUF002/RUF003`：项目包含中文与全角符号等文本场景，禁用“混淆字符”类检查以避免引发大量中文相关的误报。

### 2.3 目录级差异与豁免

Ruff 的差异策略通过 `per-file-ignores` 实现，常见场景：

- GUI 代码下允许通配导入相关规则（例如 `F403/F405`）。
- CLI 与工具脚本允许使用 `print`（对应 `T201`）。

**单点豁免原则**：
如果确实遇到无法避免或解决成本过高的规则误报，允许在单行末尾添加 `# noqa: <RuleCode>` 进行豁免，但需尽量附加注释说明豁免原因，不可滥用。

## 三、Pyright 策略

Pyright 配置统一收敛在 `pyproject.toml` 的 `[tool.pyright]`，并使用 `executionEnvironments` 做“单轨道 + 分区收紧/降噪”。

### 3.1 单轨道原则

- CI 与本地只运行一次：`pyright --level error`
- 严禁再引入独立的 `pyrightconfig*.json` 等作为分叉配置来源。

### 3.2 全局默认：basic + 关键错误收紧

全局使用 `typeCheckingMode = "basic"`，同时将高价值问题提升为 error（例如未使用的 import/variable、常见类型错误与 Optional 相关问题等），以确保门禁具备确定性且类型清晰。

### 3.3 GUI 分区：针对性降噪

GUI 目录采用独立执行环境（`root = "src/docwen/gui"`），并对部分规则降级。例如将 `reportAttributeAccessIssue` 从 error 调整为 warning，这有效降低了 GUI 代码中动态属性访问场景（如 Tkinter / ttkbootstrap 动态注入控件）的噪声。

### 3.4 桩文件（Typings / Stubs）补充

- **目录位置**：`typings/`
- **目的**：为缺少类型提示的第三方库（如 `ttkbootstrap`、`tkinterdnd2`）提供补充声明（stub）。
- **原则**：当引入新的不受类型检查支持的库导致报错时，优先考虑在 `typings/` 下补充简易 stub 以消除 `Any` 的蔓延，而不是通过全局配置关闭规则。

**类型豁免原则**：
如果特定代码的类型无法推断且不具备调整空间，可以使用 `# type: ignore`，并配合强转 `cast` 使用，以此限定影响范围。

## 四、CI 对齐要求

- CI 只能调用与本地一致的检查入口与参数。
- Pyright 不拆分 core/GUI 两次执行，统一单次运行覆盖全量检查范围。
- 保证本地门禁脚本（`tools/qa.py`）通过时，CI 同样必然通过（0 容忍任何由配置隔离导致的漂移）。
